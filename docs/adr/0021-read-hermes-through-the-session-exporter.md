# Read Hermes through the session exporter

The Hermes adapter runs beside one explicitly configured Hermes profile and reads sessions through the documented `hermes sessions export -` command. It supports only the probed v0.11.x contract: newer Hermes releases changed active and compacted message behavior without exposing inactive history through the documented CLI, so unknown versions fail closed instead of falling back to SQLite, legacy JSONL, or Python internals.

The collector consumes exporter stdout incrementally and never writes or logs the raw export. Each JSONL object is immediately reduced to minimal lineage metadata, rewrite booleans, current-Review-Day fresh Visible Messages, and a candidate cursor; it does not retain a projected copy of every exported session. It keeps only human and primary-agent text, strips Hermes reasoning and leaked tool markup, and excludes system/tool messages, `cron` and `tool` sources, the current review session, and children of live parents. User-created branches and compression continuations remain eligible when the parent was ended before the child started.

Normalization uses explicit role and content-block allowlists. An eligible session with a non-array message collection, an unknown candidate role, or an unknown candidate content block fails the complete Review Window before any cursor candidate can be returned; known excluded automation and ineligible lineage do not block collection. The fixture builder uses the same allowlist as live export.

Exporter lines remain byte-bounded across chunk boundaries. Stderr is fully drained without retaining its body, and non-zero exits, stream failures, and cancellation failures return stable errors that contain no stderr, raw session text, credential material, or local paths.

Review Day filtering and candidate cursors remain platform-independent. Hermes integer message IDs are ordered numerically when timestamps tie. If a saved cursor anchor disappears while newer rows exist, collection stops as a possible `/retry` or `/undo` rewrite rather than admitting duplicated history.

One selected Hermes profile or explicit home is one Agent Source. Setup records that boundary in a mode-`0600` Agent Blog configuration and installs only the repo-owned `agent-blog-review` skill into that home; it never directly reads or changes Hermes credentials, provider/model configuration, memories, or unrelated settings. Dry-run performs the same read-only compatibility, redacted Hermes status, private-state ignore, and repository write-access checks as apply but writes no files or schedules. Mutation requires explicit `--apply` authorization plus an operator `--confirm-repo-scope` declaration, because repository `viewerPermission` proves write access but cannot prove that the active credential has no authority outside the Publication Repository.

Live collection requires exactly one explicit boundary from the private configuration. For `hermesHome`, every version, capability, and export process receives that value as `HERMES_HOME`; for `hermesProfile`, every process receives the global `--profile <name>` arguments and any inherited `HERMES_HOME` is removed. The collector does not infer or inspect a default Hermes home, and it rejects missing or ambiguous boundaries before spawning a process.

Manual and scheduled generation use one-shot, non-resumed Hermes queries tagged `source=tool`, reuse the selected profile's existing model configuration, and enter the same shared `npm run review` lifecycle. Hermes cron is optional and gateway-dependent; an OS scheduler may invoke the same worker from the main checkout. Both paths keep the Review Skill behind the normalized private Review Window and exclude their own `tool` or `cron` activity from later reviews.
