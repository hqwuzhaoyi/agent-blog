# Capture Claude Code Visible Messages prospectively

The Claude Code adapter treats supported hook delivery as the authoritative source of timestamped Visible Messages. `UserPromptSubmit` supplies direct Agent Operator prompts, while `MessageDisplay` supplies only assistant text rendered to the operator. Display batches are normalized, deduplicated by opaque session, message, and batch identity, and committed once after every batch through the final marker is present. A second final marker with a different index is a conflicting delivery and makes coverage incomplete instead of replacing the first final index. Tool activity, thinking, images, results, synthetic inputs, subagent activity, malformed input, empty text, unknown events, and the review worker's own session are excluded before persistence.

Capture writes only normalized text, opaque identifiers, local receipt timestamps, append-only sequence values, incomplete coverage state, and minimum reconciliation metadata to a permission-restricted private journal. An accepted event's sequence is never renumbered; a later receipt with an earlier timestamp is appended with the next sequence, so prior cursors remain stable. It never stores raw hook payloads or parses Claude Code transcripts. Atomic replacement and an exclusive journal lock prevent concurrent hook processes from losing accepted events; corrupt journals, interrupted writes, conflicting batches, and uncertain provenance fail closed and cannot be presented as complete coverage.

The documented `UserPromptSubmit` contract has no invocation or delivery ID. Direct prompt identity is therefore scoped to the narrowest observable turn boundary: session, normalized prompt, and the last accepted assistant sequence. A retry before another assistant message is idempotent, while the same prompt submitted after an intervening assistant message remains a distinct event. Two identical submissions before any assistant display are contractually indistinguishable and share the boundary; capture does not invent a timing heuristic or inspect transcript contents to guess.

Coverage begins when the hooks are installed. Historical Claude Code sessions are not backfilled as exact Review Windows because the supported history contract lacks documented per-message timestamps and sufficient provenance. Hooks capture events only; a separate local scheduled task or OS scheduler later turns a complete interval into a Daily Review.

Collection accepts either that validated journal or the same closed normalized event schema in memory, plus normalized Agent SDK inventory and explicit coverage reconciliation. A Review Day is complete only when trusted start and end markers span the entire configured local day, no journal or reconciliation gaps remain, every captured session is inventoried as primary, and reconciliation reports `matched`. SDK summaries, paths, activity times, and message bodies are not Review Window inputs; only the adapter receipt timestamp assigns a Review Day.

Complete intervals return the shared `{ sourceId, reviewDay, timeZone, messages, candidateCursors }` contract. Events are ordered by their deterministic local receipt sequence and cursors retain sequence, receipt timestamp, and stable event identity per session. Missing history or markers, corrupt or partial journals, pending display batches, uncertain provenance, inventory mismatch, and missing cursor anchors return an explicit incomplete result with no messages or candidate cursors. A complete interval with no new Visible Messages returns the standard empty Review Window so the shared no-update workflow can finish without publication.

Setup requires Claude Code 2.1.166 or newer and pins `@anthropic-ai/claude-agent-sdk` to tested version 0.3.217. Dry-run performs read-only compatibility, exit-status-only authentication, ignore-rule, and repository-write-access probes. GitHub's `viewerPermission` cannot prove credential scope, so dry-run and apply also require an explicit operator confirmation that the credential is restricted to the Publication Repository and record that assertion honestly. Explicit apply structurally merges project settings, adds capture-only `UserPromptSubmit` and `MessageDisplay` hooks, preserves unrelated hooks and skills, records the initial coverage marker once, and writes private configuration with restrictive permissions.

The runtime paginates only the supported SDK session helpers and projects their results to session identity and conservative provenance. SDK summaries, paths, raw messages, and activity times are discarded. Desktop or an OS scheduler supplies the clock; setup only describes or generates the local schedule and never turns a hook into a review trigger. Every review worker starts in a dedicated Claude process with `AGENT_BLOG_CLAUDE_REVIEW_WORKER=1`; hook subprocesses inherit that process-level marker and exclude the entire worker before private journal access. The Review Skill stops when the marker was not present before startup and never relies on setting a session ID after `UserPromptSubmit` has already fired.

The live adapter snapshots the normalized journal once. Only an installation marker at or before the Review Day start, a trusted run time at or after the day end, and a snapshot without gaps or pending batches may form complete coverage. For each session with accepted events, SDK documented-message count must equal accepted journal-event count. The public SDK cannot reliably correlate every historical message with a hook delivery or exact day, so any mismatch is incomplete; the adapter never substitutes text matching, session activity, or filesystem times. Fixture mode accepts only closed normalized events, inventory, and coverage.
